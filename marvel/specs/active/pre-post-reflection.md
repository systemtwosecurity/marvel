# Pre/Post-Reflection System for Hooks

**Owner:** MARVEL Core
**Status:** Active
**Created:** 2026-03-05

**Packs Required:**
- pack:code-quality@latest
- pack:testing@latest

---

## 1. Goal

Add structured PreReflection and PostReflection to the MARVEL hooks system, enabling a prediction-validation learning loop. Before execution, MARVEL records assumptions, risks, and a plan. After execution, MARVEL compares actual outcomes against those predictions, explicitly validating or invalidating assumptions. This closes the feedback loop from "what did we think would happen" to "what actually happened" — producing higher-signal lessons than the current correction-only approach.

The reference implementation exists in `~/detections/detectionsai/worktrees/main/marvel/tools/agent-runner/` but was designed for autonomous spec execution, not interactive Claude Code sessions. This spec adapts the concept to the hooks lifecycle.

## 2. Non-Goals

- **Agent-runner port** — We are NOT porting the agent-runner module. We're adapting the reflection concept to the existing hooks system.
- **ML feature extraction** — The reference implementation has `CriticFeatures` and `ConsistencyDeltas` for training a critic model. That pipeline is out of scope. We design storage to be forward-compatible with it.
- **LLM-generated reflections** — v1 uses template/heuristic-based generation (like the reference). LLM-based generation can be layered on later.
- **Spec execution** — This does not add the ability to run specs autonomously. It adds reflection to interactive sessions.
- **JSON Schema validation** — The reference has `REFLECTION_SCHEMA.json` with AJV contract tests. We'll add TypeScript types and runtime validation but skip JSON Schema for v1.

## 3. Context

### What exists today

The hooks system has a **guidance-based** learning loop:

```
user correction → guidance.jsonl → stop hook correlates with injections
→ lesson-outcomes.jsonl → utility_score update → injection priority
```

This is reactive — it only learns when the user explicitly corrects the agent. It cannot learn from:
- Verification failures (lint/test/build) that the agent self-corrects
- Assumptions that turned out to be wrong but didn't cause a user correction
- Plans that succeeded and should reinforce confidence

### What the reference implementation has

The `agent-runner` defines a `Reflection` type with `phase: 'pre' | 'post'`:

- **PreReflection** — plan, assumptions, risks, unknowns, confidence, expectedVerification
- **PostReflection** — actualOutcome, analysis, assumptionsValidated, assumptionsInvalidated, nextSteps

These are generated by template-based heuristics in `runner/reflection.ts` and stored as `reflection-pre.json` / `reflection-post.json` in run directories. A `TraceWriter` handles validation and persistence. A feature extractor converts reflection pairs into ML training features.

**Key gap:** The reference's `REFLECTION_SCHEMA.json` explicitly states: *"PLANNED - NOT IMPLEMENTED. TraceWriter supports this format but reflections are not currently being written by the hooks system."*

### Vacant slots in current types

- `RunState.currentTask` — defined but never populated. Perfect for task-level reflection tracking.
- `RunState.lastReflectionAt` — exists but only used for guidance-based reflection timing.
- `EnhancedLesson`, `PotentialLesson`, `SessionStats` — defined in `types.ts` but unused. Forward-looking types that align with richer reflection.

## 4. Constraints and Guardrails

- **No new external dependencies** — use existing Node.js APIs and MARVEL infrastructure
- **Hooks remain non-blocking** — reflection generation must not add perceptible latency to hook responses. Write operations happen async where possible.
- **Backwards-compatible run directories** — existing runs without reflection files continue to work
- **camelCase everywhere** — the reference has a camelCase/snake_case mismatch between TypeScript and JSON Schema. We use camelCase consistently since we're skipping JSON Schema.
- **Reflection is per-task, not per-session** — a session may contain multiple tasks. Each task gets its own pre/post reflection pair. Fall back to session-level reflection if no explicit task boundaries are detected.

## 5. Proposed Change

### 5.1 Design

#### Task lifecycle and reflection triggers

```
User prompt: "Add retry logic to the API client"
  ↓
user-prompt-submit detects task_start pattern
  ↓ generates PreReflection
  ↓ writes reflection-pre-<taskId>.json to run dir
  ↓ stores activeReflection in RunState
  ↓
[execution: file reads, edits, bash commands]
  ↓
post-tool-use tracks verification outcomes (lint pass, test pass, etc.)
  ↓
Task ends via one of:
  (a) user-prompt-submit detects task_end pattern → PostReflection
  (b) stop hook fires (session ends) → PostReflection for any open task
  (c) user-prompt-submit detects new task_start → close previous task first
  ↓
PostReflection compares against PreReflection:
  - Which assumptions were validated?
  - Which were invalidated?
  - What was the actual outcome?
  ↓ writes reflection-post-<taskId>.json
  ↓ feeds assumption validation results into lesson candidates
```

#### Reflection types

```typescript
// New types in types.ts

export interface Risk {
  risk: string;
  mitigation?: string;
  likelihood?: 'low' | 'medium' | 'high';
}

export interface ActualOutcome {
  success: boolean;
  summary: string;
  failureReason?: string;
  failureStage?: 'lint' | 'test' | 'build' | 'typecheck' | 'boundary' | 'other';
}

export interface Reflection {
  phase: 'pre' | 'post';
  taskId: string;
  timestamp: string;
  plan?: string[];
  assumptions?: string[];
  risks?: Risk[];
  unknowns?: string[];
  confidence?: number;
  expectedVerification?: string[];
  actualOutcome?: ActualOutcome;
  analysis?: string;
  nextSteps?: string[];
  assumptionsValidated?: string[];
  assumptionsInvalidated?: string[];
}

export interface PreReflection extends Reflection {
  phase: 'pre';
  plan: string[];          // required
  assumptions: string[];   // required
  confidence: number;      // required (0-1)
}

export interface PostReflection extends Reflection {
  phase: 'post';
  actualOutcome: ActualOutcome;  // required
  preReflectionRef: string;      // filename of corresponding pre-reflection
}

export interface TaskReflectionState {
  taskId: string;
  description: string;
  startedAt: string;
  preReflection: PreReflection;
  verificationResults: VerificationResult[];
  filesModified: string[];
  toolCallCount: number;
  correctionCount: number;
}

export interface VerificationResult {
  type: 'lint' | 'test' | 'build' | 'typecheck';
  passed: boolean;
  timestamp: string;
}
```

#### PreReflection generation (heuristic-based)

When a task_start is detected, gather context:
1. **User's task description** — the prompt text
2. **Active packs** — which packs are loaded
3. **Recent file context** — files mentioned in the prompt or recently edited
4. **Pack guardrails** — relevant constraints

Generate assumptions based on:
- "Existing code follows established patterns" (from code-quality pack)
- "Test suite is comprehensive and passing" (from testing pack)
- "Required dependencies are available" (if imports/packages mentioned)
- Pack-specific assumptions derived from guardrails keywords

Calculate confidence starting at 0.7, adjusted by:
- Risk factors: auth/security files (-0.1), migration files (-0.1), unfamiliar patterns (-0.05)
- Positive signals: well-tested area (+0.05), recent successful similar tasks (+0.05)
- Pack coverage: more relevant packs = higher confidence (+0.02 per pack, max +0.08)

#### PostReflection generation

When a task ends, compare against the PreReflection:

1. **Collect outcomes:**
   - Verification results tracked by `post-tool-use` (lint/test/build/typecheck passes and failures)
   - Correction count during the task (from guidance)
   - Files modified (from tool_calls.jsonl)

2. **Validate assumptions:**
   - Map verification failures to specific assumptions (e.g., test failure → "Test suite is comprehensive and passing" invalidated)
   - Map user corrections to pattern-related assumptions
   - Remaining assumptions without contradicting evidence → validated

3. **Calculate confidence delta:**
   - Success with no corrections: +0.15
   - Success with corrections: +0.05
   - Failure in verification: -0.2 per unique failure stage
   - Many corrections (3+): -0.1

4. **Generate analysis** — template-based summary of what happened and why

#### Hook participation matrix

| Hook | Role in Reflection |
|------|-------------------|
| **session-start** | Initialize `activeReflection: null` in RunState. Load any incomplete reflection from previous compaction. |
| **user-prompt-submit** | Detect task_start → generate PreReflection, store in RunState and write to disk. Detect task_end → trigger PostReflection. Detect new task_start → close previous task's reflection first. |
| **pre-tool-use** | If `activeReflection` exists, include key assumptions in injection context as `<marvel-reflection>` block so the agent is aware of its own stated assumptions. |
| **post-tool-use** | Track verification outcomes (bash commands matching lint/test/build patterns). Record files modified. Feed into `activeReflection.verificationResults`. |
| **post-tool-use-failure** | Record tool failures. A verification failure (test/lint/build) is a strong signal for assumption invalidation. |
| **stop** | If `activeReflection` is open, generate PostReflection. Include reflection summary in the stop message alongside existing promotion candidates. Feed invalidated assumptions into lesson candidates. |
| **pre-compact** | Serialize `activeReflection` state into the compaction summary so it survives context compression. |
| **lifecycle hooks** | No changes needed. |

#### Learning loop integration

The PostReflection feeds into the existing learning loop at two points:

1. **Lesson candidate generation** (in stop hook): Invalidated assumptions become lesson candidates. Example: if "Existing code follows established patterns" was invalidated because lint failed, generate a candidate lesson about the specific pattern that was violated.

2. **Utility score adjustment** (in stop hook): Lessons that were injected during a task where all assumptions were validated get a utility boost. Lessons injected during tasks with invalidated assumptions get a utility reduction (they didn't prevent the issue).

```
PreReflection.assumptions
  ↓ compared with
PostReflection.assumptionsValidated / assumptionsInvalidated
  ↓ mapped to
Injected lessons during task window
  ↓ adjusts
Lesson utility_score (existing mechanism in updateLessonUtilityScores)
```

#### Storage

Per run directory:
```
marvel/runs/<runId>/
  reflection-pre-<taskId>.json    # PreReflection
  reflection-post-<taskId>.json   # PostReflection
  run.json                        # RunState now includes activeReflection
```

Task IDs are derived from timestamp: `task_<HHmmss>` (unique within a run since tasks don't overlap).

## 8. Rollback Plan

The reflection system is additive. To roll back:
1. Remove `activeReflection` from `RunState` (or leave it as optional, always `undefined`)
2. Remove reflection generation calls from hooks
3. Existing run directories with reflection files are harmless — they're just extra JSON files

No other MARVEL functionality depends on reflections, so removal has zero blast radius.

## 9. Notes for MARVEL Runner

- **Risk level:** medium
- **Key risk:** Task boundary detection via regex is imperfect. Users don't always say "let's do X" to start a task. Mitigate by also supporting session-level reflection (one pre at session start based on first substantive prompt, one post at stop) as a fallback when no explicit task boundaries are detected.
- **Key risk:** PreReflection generation adds latency to `user-prompt-submit`. Mitigate by keeping generation heuristic-based (no LLM calls) and writing files async.
- **Key risk:** Assumption validation is heuristic — mapping "test failure" to "Test suite is comprehensive" is a rough proxy. This is acceptable for v1; LLM-based analysis can improve mapping later.
- **Ordering dependency:** `post-tool-use` must record verification results BEFORE `stop` generates the PostReflection. This is guaranteed by Claude Code's hook lifecycle.
- **Reference code:** The type definitions and generator logic in `~/detections/detectionsai/worktrees/main/marvel/tools/agent-runner/src/types/reflection.ts` and `runner/reflection.ts` should be adapted, not copied verbatim. The camelCase/snake_case mismatch must be resolved (use camelCase). The `TracedReflection` wrapper and `tokenCount` field are unnecessary for hooks.
- **Forward compatibility:** Store reflection files as pretty-printed JSON so they're human-readable and future feature extractors can parse them. Include a `version: 1` field in each reflection file for schema evolution.
